Regex, heuristiques et extraction de documents dans Paperflow

Le principe du projet est simple : on lit un document avec l'OCR, on nettoie le texte, puis on extrait des blocs utiles avec deux outils complémentaires : les regex et les heuristiques.

Regex

Décrire la forme d'une donnée : date, SIREN, email, IBAN, total TTC.

Heuristique

Déduire une zone probable selon le contexte, l'ordre ou la position dans le document.

OCR

Produit des tokens rec_texts qui ne sont pas toujours parfaits.

JSON final

Contient les blocs extraits et les groupes nommés utiles au métier.

C'est quoi une regex ?

Définition simple

Une regex, ou expression régulière, est un motif de recherche. On ne cherche pas une valeur exacte, on cherche une forme.

  • \d{9} signifie "exactement 9 chiffres".
  • \d{14} signifie "exactement 14 chiffres".
  • [A-Z]{2}\d{2} correspond au début d'un IBAN comme FR76.

À quoi ça sert ici ?

  • Reconnaître un SIREN, un SIRET, une date, un BIC, un email.
  • Délimiter un bloc fixe comme Greffe du Tribunal... ou Cordialement.
  • Capturer plusieurs morceaux dans le même motif grâce aux groupes.

Exemple concret

Siret\s*:\s*(\d{14})
  • Siret = texte fixe.
  • \s* = zéro ou plusieurs espaces.
  • : = deux-points attendu.
  • (\d{14}) = on capture les 14 chiffres du SIRET.

C'est quoi une heuristique ?

Définition simple

Une heuristique est une règle pratique. Elle n'est pas mathématiquement parfaite, mais elle marche bien dans la plupart des cas réels.

Pourquoi on en a besoin

  • Les documents ne sont pas toujours alignés pareil.
  • L'OCR peut couper une phrase en plusieurs tokens.
  • Une regex unique peut devenir trop fragile si la mise en page varie.

Exemples dans ce projet

  • Sur une facture, prendre les lignes entre Description et le deuxième Total HT.
  • Sur un RIB, prendre la zone entre Domiciliation et IBAN.
  • Si un token ressemble à 34000 Montpellier, le traiter comme ville + code postal.
À dire : une regex cherche une forme précise, une heuristique cherche une zone probable ou une logique de lecture du document.

Règles de base des regex

Classes de caractères

  • \d = un chiffre.
  • \s = un espace, une tabulation ou un retour à la ligne.
  • [A-Z] = une majuscule.
  • [A-Za-z0-9] = lettre ou chiffre.
  • . = n'importe quel caractère.

Quantificateurs

  • + = un ou plusieurs.
  • * = zéro ou plusieurs.
  • ? = optionnel.
  • {9} = exactement 9 fois.
  • {2,4} = entre 2 et 4 fois.

Groupes

  • (...) = groupe capturé.
  • (.+?) = capture un texte variable sans trop déborder.
  • Dans ces scripts, les groupes deviennent les champs du JSON final.

Exemple décomposé

Num[ée]ro de facture\s*:?\s*([A-Z0-9-]+)
  • Num[ée]ro accepte Numero ou Numéro.
  • \s* accepte les espaces éventuels.
  • :? accepte la présence ou non du :.
  • ([A-Z0-9-]+) capture le numéro de facture.

Pipeline général des scripts

1. Entrée
Image ou PDF.
2. OCR
run_ocr() produit des tokens rec_texts.
3. Nettoyage
On corrige les fautes OCR et les espaces.
4. Extraction
Regex et heuristiques récupèrent les blocs utiles.
5. JSON
On sauvegarde tout dans un fichier final.
Point important : dans ce projet, la qualité de l'extraction dépend autant de la normalisation OCR que des regex.

Comment est construit le fichier de sortie

Les scripts écrivent presque tous un JSON final de cette forme :

{
  "source_image": "...",
  "ocr_result_file": "...",
  "regex_config_file": "...",
  "rec_texts": ["token 1", "token 2"],
  "joined_text": "texte OCR nettoyé",
  "extracted_blocks": {
    "bloc_x": {
      "description": "...",
      "regex": "...",
      "matched": true,
      "full_match": "...",
      "groups": {
        "champ_1": "...",
        "champ_2": "..."
      }
    }
  }
}
  • rec_texts = morceaux bruts reconnus par l'OCR.
  • joined_text = texte concaténé et normalisé.
  • matched = vrai ou faux selon la réussite de l'extraction.
  • full_match = texte complet capturé.
  • groups = champs nommés utiles comme siren, date_facture, iban.

Regex ou heuristique : dans quel cas ?

On choisit une regex quand...

  • la forme de la donnée est stable;
  • le libellé autour est bien connu;
  • on veut capturer une valeur précise.

Exemples : date, SIREN, SIRET, IBAN, BIC, numéro de facture.

On choisit une heuristique quand...

  • la mise en page change d'un document à l'autre;
  • l'OCR segmente mal les lignes;
  • on repère mieux une zone qu'une forme exacte.

Exemples : bloc vendeur, bloc client, lignes d'un tableau, domiciliation bancaire.

En pratique

  • Document officiel stable : plutôt regex longues par sections.
  • Document commercial variable : mélange regex + heuristiques.
  • C'est exactement ce qu'on voit dans les 6 fichiers.

Logique par fichier

URSSAF

test_local_regex_vigilance_urssaf.py

Document très structuré : ici on est surtout sur de grosses regex par blocs.

  • Nettoyage OCR ciblé : villes mal lues, mots déformés, espaces incohérents.
  • Extraction de sections fixes : émetteur, destinataire, cadre légal, message, signature.
  • Usage fort de \s+ et (.+?) pour absorber les retours ligne OCR.
Exemple :
([A-Z]+,?le\s+[0-9]{2}/[0-9]{2}/[0-9]{4})

But :
capturer une ville en majuscules + "le" + une date.

Pourquoi regex ? Parce que le courrier URSSAF suit une structure très répétitive.

RIB

test_local_regex_rib.py

Cas typique où la regex seule ne suffit pas, donc le script mélange regex et heuristiques.

  • DOC_TYPE_RE détecte le type de document.
  • IBAN_RE et BIC_RE reconnaissent des formats universels.
  • Le bloc RIB national est reconstruit en cherchant des tokens de bonne longueur.
  • Le titulaire est reconstruit par ordre logique : nom, adresse, ville, pays.
Exemples :
IBAN_RE = [A-Z]{2}\d{2}[A-Z0-9]{11,30}
BIC_RE  = \b[A-Z0-9]{8,11}\b

Heuristique :
prendre la zone entre "Code banque" et "IBAN".

Pourquoi heuristique ? Parce qu'un RIB change selon la banque, donc la mise en page n'est pas toujours stable.

KBIS

test_local_regex_kbis.py

Approche très orientée blocs administratifs complets.

  • Le script extrait le greffe, l'identification de la société, les dirigeants, l'activité et les observations.
  • Les regex s'ancrent sur des titres très stables comme IDENTIFICATION DE LA PERSONNE MORALE.
  • Chaque capture est ensuite transformée en champs nommés.
Exemple :
Greffe du Tribunal de Commerce de\s+(.+?)\s+Code de vérification

Le groupe (. +?) capture la ville du greffe
jusqu'au prochain libellé fixe.

Pourquoi regex ? Parce qu'un KBIS officiel a des titres très réguliers.

FACTURE

test_local_regex_facture.py

Script hybride : regex pour les métadonnées stables, heuristiques pour les zones variables.

  • Nettoie beaucoup le bruit OCR.
  • Déduit vendeur et client selon la position des blocs avant le mot Facture.
  • Extrait numéro, date et numéro client avec une regex stable.
  • Repère les lignes entre Description et le deuxième Total HT.
Exemple regex :
Num[ée]ro de facture\s*:?\s*([A-Z0-9-]+)

Exemple heuristique :
prendre les tokens entre "Description"
et le deuxième "Total HT".

Pourquoi mixte ? Les entêtes sont stables, mais le corps de facture varie plus.

DEVIS

test_local_regex_devis.py

Le devis est souvent plus libre visuellement, donc le script prévoit plus de fallbacks.

  • Regex de base : téléphone, date, montant, email, code postal.
  • Le vendeur et le client peuvent être extraits soit par libellés, soit par colonnes OCR.
  • Les infos du devis tentent d'abord une regex, puis passent en heuristique si besoin.
  • Le tableau est pris entre l'entête de colonnes et le début des totaux.
Exemples :
PHONE_RE = (?:\+?\d[\d .()/:-]{7,}\d)
DATE_RE  = \b\d{1,2}[./-]\d{1,2}[./-]\d{2,4}\b

Heuristique :
si la regex globale échoue,
on reconstruit les infos morceau par morceau.

Pourquoi plus d'heuristiques ? Parce que le devis varie beaucoup selon l'émetteur.

ATTESTATION SIRET

test_local_regex_attestion_siret.py

Comme le KBIS, c'est un document officiel, donc la stratégie repose surtout sur les sections fixes.

  • Le script nettoie plusieurs erreurs OCR avant extraction.
  • Il extrait l'entête, l'identité, le siège, la gestion, la qualité du dirigeant et l'établissement.
  • Les groupes donnent ensuite siren, siret, code_ape, capital_social, etc.
Exemple :
Siret\s*:\s*(\d{14})

Ici la regex est parfaite,
car le format SIRET est très stable.

Pourquoi regex ? Parce que le document administratif est très codifié.

Script oral prêt à dire

Dans ce projet, on utilise deux idées principales : les regex et les heuristiques. Une regex sert à décrire la forme d'une donnée, par exemple une date, un SIREN ou un IBAN. Une heuristique, au contraire, est une règle pratique qu'on utilise quand la mise en page du document varie trop pour écrire une seule regex fiable.

Le pipeline est toujours le même. On commence par faire un OCR avec PaddleOCR. Ensuite, le script récupère les textes détectés dans rec_texts, puis les fusionne dans joined_text. Après cela, il applique une normalisation pour corriger les erreurs classiques de l'OCR.

Pour les documents très structurés comme le KBIS, l'attestation SIRET ou l'attestation URSSAF, on utilise surtout de grandes regex ancrées sur les titres officiels du document. Dans ce cas, le plus efficace est de capturer une section entière puis d'en extraire les champs avec des groupes.

Pour les documents plus variables comme le RIB, la facture ou le devis, on ajoute des heuristiques. Par exemple, sur une facture, on peut prendre les lignes entre Description et Total HT. Sur un RIB, on peut prendre la zone entre Domiciliation et IBAN.

Le résultat final est un JSON avec le texte OCR brut, le texte nettoyé et les blocs extraits. Chaque bloc indique si l'extraction a réussi, quel texte a été capturé et quelles données métiers ont été retrouvées.

Donc l'idée générale, ce n'est pas seulement écrire des regex. C'est choisir la bonne méthode selon le document : regex si la structure est stable, heuristique si elle varie trop.
